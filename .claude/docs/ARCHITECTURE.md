# Architecture

## High-Level System Design

The system is composed of two primary layers: the **Knowledge Layer** (Obsidian) and the **Reasoning Layer** (MAGI).

```mermaid
graph TD
    User[User] -->|Writes/Reads| KB[Knowledge Base (Obsidian)]
    User -->|Defines Problem| Inputs[Input Files (.md)]

    subgraph MAGI System
        Orchestrator[Pipeline Orchestrator]
        IssueEngine[Issue-Driven Engine]
        HypothesisEngine[Hypothesis-Driven Engine]

        Orchestrator --> IssueEngine
        Orchestrator --> HypothesisEngine
        IssueEngine -->|Critical Issue| HypothesisEngine
    end

    Inputs --> Orchestrator
    MAGI System -->|Generates| Artifacts[Output JSON/MD]
    Artifacts -->|Visualized in| KB
```

## Component Details

### 1. Knowledge Base (`knowledge-base/`)

* **Purpose**: Long-term memory and user interface.
* **Format**: Markdown files structured by PARA or Zettelkasten methods.
* **Role**: Serves as the source of context and the destination for high-quality insights generated by MAGI.

### 2. MAGI System (`magi_system/`)

A file-based agent orchestration framework.

#### A. Pipeline Controller (`magi_system/pipeline/`)

* **Role**: Master Supervisor.
* **Logic**:
    1. Checks state (existence of output files).
    2. Determines the next required stage.
    3. Prepares context (bridging outputs from previous stages to inputs for the next).
    4. Instructs the external Agent (User/LLM) to execute the specific prompt.

#### B. Issue Engine (`magi_system/issue/`)

* **Based on**: "Issue Thinking" (Ron-ten Shikou).
* **Goal**: Disambiguation.
* **Stages**:
  * Definition -> Structuring (MECE) -> Selection (Impact/Control) -> Integration.

#### C. Hypothesis Engine (`magi_system/hypothesis/`)

* **Based on**: "Hypothesis Thinking" (Kasetsu Shikou).
* **Goal**: Actionable Solution.
* **Stages**:
  * Framing -> Structuring -> Divergence (Casper) -> Test (Melchior) -> Convergence.

### 3. Integration (`langfuse_integration/`)

* **Purpose**: Quality control and observability of the LLM interactions (if enabled).

## Data Flow

1. **Input**: User edits a Markdown file (e.g., `mission.md`).
2. **Process**: Python scripts read the file -> Construct Prompt -> LLM Execution (External) -> Save JSON.
3. **State**: JSON files in `output/` directories serve as checkpoints.
4. **Feedback**: User reads JSON/Markdown output -> Updates `feedback.md` -> Pipeline reiterates.
